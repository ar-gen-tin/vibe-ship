# vibe-ship — 纯粹 Vibe 少踩坑

给不懂代码的 Vibe Coder 的开发引导流程。不教你写代码，教你按什么顺序做事。

这个技能从三个真实项目、60+ 个 bug、两次技术栈推倒重来、一次删库重建中提炼出来。

## Usage

```
/vibe-ship          # 从头开始，或自动从上次停的地方继续
/vibe-ship --status # 查看当前进度
```

启动时自动读取项目根目录的 `VIBE-SHIP.md`，如果存在就从第一个未完成的步骤继续。不存在就从头开始。

## 七步流程

```
想清楚 → 问技术栈 → 最小版本 → 发布(含安全扫描) → 做设计 → 砍功能 → 记录一切
  [1]      [2]        [3]            [4]              [5]       [6]       [7]
```

## 进度文件 VIBE-SHIP.md

在项目根目录创建或更新，作为整个流程的单一状态文件：

```markdown
# vibe-ship 进度

## 产品定义
- 给谁用：
- 核心功能：
- 最像的产品：

## 技术方案
- 方案：
- 原因：

## 进度
- [ ] Step 1: 想清楚
- [ ] Step 2: 技术栈
- [ ] Step 3: 最小版本
- [ ] Step 4: 发布（含安全扫描）
- [ ] Step 5: 设计
- [ ] Step 6: 功能审查
- [ ] Step 7: 踩坑记录

## 以后再说清单
| 功能 | 来源步骤 | 最终决定 |
|------|----------|----------|

## 踩坑记录
| 日期 | 问题 | 解决方法 |
|------|------|----------|
```

每完成一步更新进度。开发过程中遇到的坑随时记到「踩坑记录」部分，不要等到最后。

## Step 1: 想清楚再开口

在写任何代码之前，把三个问题想明白。

问用户以下三个问题（一次问完）：

1. **这个东西做出来给谁用？**
   - 提示：自己用？给朋友用？放到网上让任何人用？

2. **核心功能是什么？只说一个。**
   - 提示：如果只能做一件事，是什么？"能播放音乐"、"能显示天气"、"能追踪价格"这种一句话。

3. **你见过最像的产品是什么？**
   - 提示：可以是一个 App、一个网站、一张截图、一个链接。没有的话说"没有"也行。

收到回答后：
- 整理成一段话，复述给用户确认
- 保存到 VIBE-SHIP.md 的「产品定义」部分
- 告诉用户：「这三个答案就是你的产品定义，后面想加功能的时候回来看一眼。」

**完成信号**：用户确认了三个答案。

---

## Step 2: 问技术栈

在写代码之前，确认技术方案。

问用户两个问题：

1. **你想做什么类型的程序？**
   - 选项：手机 App / 桌面应用 / 网页 / 浏览器插件 / 命令行工具 / 其他

2. **你的电脑是什么系统？**
   - 选项：macOS / Windows / Linux

收到回答后：
- 根据产品定义和平台，推荐 2-3 个技术方案
- 每个方案用一句话说清楚：是什么、为什么适合、有什么缺点
- 给出明确推荐，不要说"都可以"
- 用用户能听懂的话解释，听不懂就换个说法

用户选定后，保存到 VIBE-SHIP.md 的「技术方案」部分。

**完成后**：帮用户初始化项目仓库（git init + 首次提交）。用简单的话解释："这一步是给你的项目建一个存档点，以后随时可以回到这个状态。"

**完成信号**：用户选定方案，仓库已初始化。

---

## Step 3: 最小版本先跑起来

做出最丑但能用的 v0.1。

先确认：

> 回顾一下你定的核心功能："[复述核心功能]"。现在我们只做这一个功能，不做设计、不做多语言、不做动画、不做任何额外的东西。确认开始？

开发过程中的规则：
- 只实现核心功能
- 用户说"能不能顺便加个 XXX"时，回复：「记下来了，放到 Step 6 再说。现在先把核心功能跑通。」
- 把额外请求记录到 VIBE-SHIP.md 的「以后再说清单」
- 遇到的问题随时记到「踩坑记录」

### 卡住了怎么办

Vibe Coder 最常见的状态就是卡住。如果代码跑不起来：
- 让用户把屏幕上看到的错误信息发过来（截图或复制粘贴都行）
- 尝试修复，如果 5 分钟内修不好，换个方案实现同样的功能
- 如果反复卡在同一个技术上，回到 Step 2 考虑换技术栈，沉没成本不要管

### 代码审查

核心功能跑通之后，建议用户做一次代码质量检查：

> "你的程序能跑了。在继续之前，建议跑一次代码审查，检查有没有隐藏的问题。输入 `/code-review` 开始。这一步可选，但推荐做。"

修复审查发现的问题后再进入下一步。

**完成信号**：核心功能能跑了。提醒用户截个图存档。

---

## Step 4: 发布（含安全扫描）

确认这个程序可以发给别人用。这一步包含安全检查，两件事一起做。

### 4a. 安全扫描（必做，不可跳过）

在推送任何代码到 GitHub 之前，扫描项目：
- API key / secret / token
- 个人邮箱地址
- 本地电脑路径（/Users/xxx、C:\Users\xxx）
- 密码、私钥文件
- 检查 `.gitignore` 是否包含 `.env`、`*.key`、`credentials.*`

把扫描结果告诉用户：

> "推送之前，我帮你做了一次安全检查。[结果]。确认处理完再继续。"

如果已经推送了敏感信息，明确告诉用户：删文件不够，历史记录里还有，需要清除整个历史。然后自动帮用户处理，不要只说工具名让用户自己查。

### 4b. 走通发布流程

问用户：

> "代码安全检查通过了。你打算怎么分发这个程序？"

选项（根据技术栈调整）：
- 发给朋友直接安装
- 放到 GitHub 让人下载
- 上架应用商店
- 部署成网站

根据用户选择和 Step 2 的技术栈，动态生成该平台的发布检查清单，用用户能听懂的话解释每一步，然后带用户逐项完成。

通用检查项：
- 版本号统一（只在一个地方定义）
- README 写了吗（至少说清楚这是什么、怎么安装）

**完成信号**：安全检查通过 + 用户成功把程序发给了至少一个人（或部署成功）。

---

## Step 5: 现在可以做设计了

在功能稳定之后才做设计。

问用户：

1. **你有喜欢的参考图或参考产品吗？**
   - 截图、链接、或者描述风格都行。

2. **给你出几个设计方向？**
   - 默认 2 个，最多 3 个。用户说"多来几个"时回复：「3 个封顶，选择太多反而会卡住。」

设计过程中的规则：
- 最多出 3 个方案
- 用户犹豫不决时提醒：「先选一个做下去，不满意随时可以改。」
- 不要在设计阶段引入新功能

**完成信号**：用户选定了一个设计方向并实现了。

---

## Step 6: 功能审查

审查「以后再说清单」，决定哪些真的要做。

读取 VIBE-SHIP.md 里的清单。如果超过 5 条，先告诉用户：「一次最多挑 3 个做，剩下的留给 v2。」

对每个功能问：

> 「[功能名称]」— 三个选择：现在做 / v2 再做 / 不做？

在 VIBE-SHIP.md 里更新每条的「最终决定」列。

选了"现在做"的功能，按 Step 3 的规则逐个实现（只做功能，不膨胀）。

全部实现后，建议做一次全面审查：

> "新功能加完了，代码量比 v0.1 大不少。建议跑一次全面审查，让多个专家同时检查。输入 `/team` 开始。这一步可选，但如果加了不少功能的话推荐做。"

### 发布前再扫一遍

新功能实现后，重新跑一次安全扫描（和 Step 4a 同样的检查），确保新代码没有引入敏感信息。

**完成信号**：清单审查完毕，功能实现并通过安全检查。

---

## Step 7: 记录一切

整理踩坑文档，为下一个项目积累经验。

VIBE-SHIP.md 的「踩坑记录」部分在开发过程中已经持续记录了。这一步做收尾：

1. 回顾整个开发过程，问用户：「还有没有印象深刻的坑没记下来的？」
2. 补充遗漏的记录
3. 告诉用户：「下次开新项目的时候把 VIBE-SHIP.md 贴给 AI 看，它不会自己记住，但你可以替它记。」

**完成信号**：用户确认踩坑记录完整。

## 全局规则

### 语言
- 用用户的语言交流
- 技术术语必须解释，或者直接换成用户能听懂的话
- 禁止假设用户懂任何编程概念

### 节奏
- 每一步完成后明确告诉用户"第 X 步完成了，接下来是第 Y 步"
- 不要在一步里偷偷做下一步的事
- 用户想跳步时提醒：「建议按顺序来，跳过的话后面可能踩坑。但你坚持的话也行。」

### 暂停和继续
- 用户随时可以停下来，进度保存在 VIBE-SHIP.md 里
- 下次输入 `/vibe-ship` 自动从上次的位置继续

### 防膨胀
- 开发过程中用户提出的额外功能一律记录到「以后再说清单」
- 只有到 Step 6 才审查这个清单，一次最多实现 3 个
- 在 Step 3 之前不做任何设计工作
- 在 Step 4a 安全扫描之前不推送到 GitHub

### 安全
- Step 4a 安全扫描不可跳过
- 每次用户说要推送代码时，自动触发安全扫描提醒
- Step 6 新功能实现后再扫一遍

### 踩坑记录
- 开发过程中遇到的问题随时记到 VIBE-SHIP.md 的「踩坑记录」部分
- 每个坑一行：日期 + 问题 + 解决方法
- 不要等到最后靠回忆
