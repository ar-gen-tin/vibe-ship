# vibe-ship — 纯粹 Vibe 少踩坑

给不懂代码的 Vibe Coder 的开发引导流程。不教你写代码，教你按什么顺序做事。

## Usage

```bash
/vibe-ship              # 从头开始，进入第一步
/vibe-ship --step 3     # 跳到第三步（已经完成前面步骤的情况）
/vibe-ship --status     # 查看当前进度
```

## 核心理念

这个技能从三个真实项目、60+ 个 bug、两次技术栈推倒重来、一次删库重建中提炼出来。

规则只有一条：**按顺序走，别跳步。**

## 八步流程

```
想清楚 → 问技术栈 → 最小版本 → 走通发布 → 安全扫描 → 做设计 → 砍功能 → 记录一切
  [1]      [2]        [3]        [4]        [5]        [6]       [7]       [8]
```

## Workflow

### 进度追踪

在项目根目录创建或更新 `VIBE-SHIP.md` 记录进度：

```markdown
# vibe-ship 进度

- [x] Step 1: 想清楚
- [x] Step 2: 技术栈
- [ ] Step 3: 最小版本
- [ ] Step 4: 发布流程
- [ ] Step 5: 安全扫描
- [ ] Step 6: 设计
- [ ] Step 7: 功能审查
- [ ] Step 8: 踩坑记录
```

每完成一步，更新这个文件。用户用 `--status` 时读取这个文件显示进度。

---

### Step 1: 想清楚再开口

**目标**：在写任何代码之前，把三个问题想明白。

**问用户三个问题**（用 AskUserQuestion，一次性问完）：

1. **这个东西做出来给谁用？**
   - 提示：自己用？给朋友用？放到网上让任何人用？

2. **核心功能是什么？只说一个。**
   - 提示：如果只能做一件事，是什么？"能播放音乐"、"能显示天气"、"能追踪价格"这种一句话。

3. **你见过最像的产品是什么？**
   - 提示：可以是一个 App、一个网站、一张截图、一个链接。没有的话说"没有"也行。

**收到回答后**：
- 把三个答案整理成一段话，复述给用户确认
- 确认后保存到 `VIBE-SHIP.md` 的 Step 1 下面
- 提醒用户：「这三个答案就是你的产品定义，后面所有决定都围绕它。想加功能的时候回来看一眼。」

**完成信号**：用户确认了三个答案。

---

### Step 2: 问技术栈

**目标**：在写代码之前，确认技术方案。

**AI 需要做的事**：
1. 根据 Step 1 的产品定义，推荐 2-3 个技术方案
2. 用用户能听懂的话解释每个方案的优缺点
3. 明确说出推荐哪个以及为什么

**问用户的问题**（用 AskUserQuestion）：

1. **你想做什么类型的程序？**
   - 选项：手机 App / 桌面应用 / 网页 / 浏览器插件 / 命令行工具 / 其他

2. **你的电脑是什么系统？**
   - 选项：macOS / Windows / Linux

**收到回答后**：
- 列出 2-3 个方案，每个用一句话说清楚是什么、为什么适合、有什么缺点
- 给出明确推荐
- 等用户选择

**禁止**：
- 不要列超过 3 个方案
- 不要用用户听不懂的术语，听不懂就换个说法
- 不要说"都可以"，必须有推荐

**完成信号**：用户选定了技术方案，记录到 `VIBE-SHIP.md`。

---

### Step 3: 最小版本先跑起来

**目标**：做出最丑但能用的 v0.1。

**问用户一个问题**：

> 回顾一下 Step 1 你定的核心功能："[复述核心功能]"。现在我们只做这一个功能，不做设计、不做多语言、不做动画、不做任何额外的东西。确认开始？

**开发过程中的规则**：
- 只实现核心功能，拒绝所有"顺便加一个"的请求
- 如果用户说"能不能顺便加个 XXX"，回复：「记下来了，放到 Step 7 再说。现在先把核心功能跑通。」
- 把额外请求记录到 `VIBE-SHIP.md` 的「以后再说清单」里

**完成信号**：核心功能能跑了。提醒用户截个图存档。

---

### Step 4: 走通发布流程

**目标**：确认这个程序可以发给别人用。

**AI 需要做的事**：
1. 根据技术栈，列出发布需要的所有步骤
2. 用用户能听懂的话解释每一步是干什么的
3. 带用户一步步走完

**问用户**：

> 代码能跑了，但还不能发给别人用。接下来我带你走一遍发布流程。你打算怎么分发？

选项（根据技术栈调整）：
- 发给朋友直接安装
- 放到 GitHub 让人下载
- 上架应用商店
- 部署成网站

**关键检查项**（根据平台）：

macOS 应用：
- [ ] 代码签名（有没有 Apple Developer 账号？）
- [ ] 打包成 DMG 或 .app
- [ ] 如果要分发：Apple 公证

网页应用：
- [ ] 能在本地之外访问吗？
- [ ] 部署到哪里？（Vercel / Cloudflare / 其他）

通用：
- [ ] 版本号统一（只在一个地方定义）
- [ ] README 写了吗？（至少说清楚这是什么、怎么安装）

**完成信号**：用户成功把程序发给了至少一个人（或部署成功）。

---

### Step 5: 安全扫描

**目标**：确保没有把不该公开的东西推到 GitHub。

**AI 需要做的事**：
1. 扫描项目代码，检查以下内容：
   - API key / secret / token
   - 个人邮箱地址
   - 本地电脑路径（/Users/xxx）
   - 密码
   - 私钥文件
2. 检查 `.gitignore` 是否包含 `.env`、`*.key`、`credentials.*`
3. 如果发现问题，告诉用户具体是哪个文件的什么内容

**问用户**：

> 在推送到 GitHub 之前，我帮你做一次安全检查。扫描结果如下：
> [列出发现的问题，或者"没有发现问题"]
>
> 确认处理完所有问题后再推送？

**如果已经推送了敏感信息**：
- 明确告诉用户：「删文件不够，Git 会记住历史，需要清除历史记录。」
- 引导用户完成 git 历史清理（用 git-filter-repo 或 BFG）
- 提醒轮换泄露的密钥

**完成信号**：扫描通过，或问题已修复。记录到 `VIBE-SHIP.md`。

---

### Step 6: 现在可以做设计了

**目标**：在功能稳定之后做设计。

**问用户**：

1. **你有喜欢的参考图或参考产品吗？**
   - 提示：截图、链接、或者描述你喜欢的风格都行。

2. **给你出几个设计方向？**
   - 默认 2 个，最多 3 个。如果用户说"多来几个"，回复：「3 个封顶，选择太多反而会卡住。」

**设计过程中的规则**：
- 最多出 3 个方案
- 用户犹豫不决时提醒：「先选一个做下去，不满意随时可以改。」
- 不要在设计阶段引入新功能

**完成信号**：用户选定了一个设计方向并实现了。

---

### Step 7: 功能审查

**目标**：审查"以后再说清单"，决定哪些真的要做。

**AI 需要做的事**：
1. 读取 `VIBE-SHIP.md` 里的「以后再说清单」
2. 对每个功能问用户一个问题

**对每个功能问**：

> 「[功能名称]」— 这个不做，产品还能用吗？

- 用户说"能" → 标记为「不做」
- 用户说"不能"或"我想做" → 排进待办

**完成信号**：清单审查完毕，待办确认。

---

### Step 8: 记录一切

**目标**：建立踩坑文档，为下一个项目积累经验。

**AI 需要做的事**：
1. 回顾整个开发过程，整理遇到的问题
2. 创建或更新项目根目录的 `PITFALLS.md`

**格式**：
```markdown
# 踩坑记录

## [日期] [一句话描述问题]
原因：[简短说明]
解决：[怎么解决的]
```

**问用户**：

> 开发过程中有没有什么印象深刻的坑？我帮你记下来。下次开新项目的时候把这个文件贴给 AI 看，它不会自己记住，但你可以替它记。

**完成信号**：`PITFALLS.md` 创建完成。

---

## 全局规则

### 语言
- 用用户的语言交流
- 技术术语必须解释，或者直接换成用户能听懂的话
- 禁止假设用户懂任何编程概念

### 节奏
- 每一步完成后明确告诉用户"第 X 步完成了，接下来是第 Y 步"
- 不要在一步里偷偷做下一步的事
- 用户想跳步时提醒：「建议按顺序来，第 X 步跳过的话后面可能会踩坑。但你坚持的话也行。」

### 防膨胀
- 开发过程中用户提出的额外功能一律记录到「以后再说清单」
- 只有到 Step 7 才审查这个清单
- 在 Step 3 之前不做任何设计工作
- 在 Step 4 之前不推送到 GitHub

### 安全
- Step 5 不可跳过
- 每次用户说要推送代码时，自动触发安全扫描提醒
